## 1. 为什么需要多 Agent

### 一个老问题的新解法

你有没有过这样的经历：周一早上，你试图同时回复邮件、写季度报告、准备下午的会议PPT，结果到中午才发现——邮件没回完，报告写了一半，会议材料还没开始。每一个任务都只做了一点点，质量也勉强及格。

这不奇怪。一心多用本来就是人类认知的软肋。而当我们把 AI agent 作为一个"人"来看待时，它同样面临这个问题：**一个 agent 能做的事很多，但没有哪个 agent 能把所有事都做到极致。**

多 Agent 系统，正是为解决这个根本矛盾而生的架构思路。

### 单个 Agent 的天花板

#### 上下文窗口的硬性约束

现代大模型的上下文窗口虽然已经从 4K 扩展到了百万 token 量级，但这并不意味着一个 agent 能"消化"无限信息。当你在同一个对话里塞入大量参考资料、代码库、历史对话，模型的表现会逐渐下降——关键信息被稀释，注意力分散，输出质量随之滑坡。

更现实的问题是：**不同的任务需要不同的上下文。** 研究任务需要大量外部资料，分析任务需要清洗后的结构化数据，写作任务需要明确的风格指南和参考资料。把这些全部塞进一个 prompt 里，不仅 token 浪费严重，模型还容易"串台"——把研究时的严谨语气带到产品文案里，或者把数据分析的谨慎带到创意写作中。

#### 角色冲突：什么都想做，什么都做不精

让一个 agent 同时扮演研究员、分析师和写手，就像让同一个人同时做三份工作：早上做市场调研，下午处理销售数据，晚上写品牌文案。听起来效率很高，实际上每个角色都在互相拉扯。

你给 agent 的 prompt 大概是这样的：

> "你是一个专业的AI助手，需要先研究新能源汽车市场数据，然后分析竞争格局，最后写一份5000字的市场研究报告。"

这个 prompt 隐含了一个假设：同一个模型能在这三个任务上都表现出色。但实际上：

- **研究员需要批判性思维**，能判断信息可信度、发现数据异常
- **分析师需要结构化思维**，擅长归纳规律、发现趋势
- **写手需要表达力**，能把复杂结论转化为清晰的叙事

这三种思维方式放在一起，模型往往会"取平均值"——每样都做，但每样都平庸。更糟糕的是，当三个目标略有冲突时（比如研究的结论和分析的假设不一致），一个 agent 很难做出明智的取舍——它会试图同时满足所有指令，结果是输出的东西两头都不靠。

### 专业化分工：一个久经验证的智慧

将复杂任务分解为多个专业化子任务，由不同的专精 agent 分别处理——这个思路并非 AI 时代的发明，它来自一个我们都熟悉的经济学原理：**劳动分工**。

亚当·斯密在《国富论》中用别针工厂的例子说明了这一点：一个人单独生产别针，一天可能做不出一枚；但将生产过程分为抽丝、拉直、切截、削尖、磨光等十几道工序，由不同工人分别负责，专业化操作能让人均日产量达到数千枚。**专业化使效率爆炸式增长。**

多 Agent 系统的设计哲学与此一脉相承。与其打造一个"全能型" agent，不如让一群"专家型" agent 各司其职，通过协作完成复杂任务。每个 agent 有明确的职责边界、专属的工具和知识库、以及针对性的优化方向。

这不是在制造复杂性，而是在**用结构化的方式管理复杂性**。

### 多 Agent 协作的核心价值

多 Agent 架构之所以值得投入，核心在于三个价值点：

**第一，专注力。** 每个 agent 只处理一件事，可以把全部"注意力"投入其中。研究 agent 不会被"我该怎么写得吸引人"这类写作问题分心，分析 agent 不会被"语气应该正式还是轻松"这类表达问题干扰。专注带来质量。

**第二，可扩展性。** 当任务规模扩大，你不需要重新训练一个更大的模型，而是增加更多同类型的 agent 实例。研究 100 份行业报告？启动 10 个研究 agent 并行工作，而不是让一个 agent 拼命吞吐。系统整体吞吐量随 agent 数量线性扩展。

**第三，可维护性。** 每个 agent 的 prompt 和工具链是独立的。当研究方法需要升级，你只改研究 agent；写作风格需要调整，你只改写手 agent。改动的影响范围被天然隔离，降低了出错的概率。

### 具体的对比：一个报告的两种诞生方式

让我们用一个实际场景感受差异。

**方式一：单一 Agent 全程包办**

你给 agent 发送指令："帮我写一份2024年中国新能源汽车市场竞争格局的研究报告，10000字，需要包含市场数据、竞争对手分析、趋势预测和建议。"

这个 agent 需要自己完成：搜索最新市场数据 → 整理数据 → 分析竞争格局 → 构思报告结构 → 撰写 → 润色。整个过程在一个"大脑"里完成，上下文窗口被大量原始数据占据，写作阶段已经很难"记得"研究阶段的细节，数据引用可能出错，逻辑连贯性也难以保证。

**方式二：三个专业 Agent 协作**

```
[研究 Agent] → 负责搜集市场数据、竞品信息、行业报告
              ↓ 输出结构化的数据摘要和关键发现

[分析 Agent] → 负责深度分析：竞争格局、SWOT、趋势归纳
              ↓ 输出有洞察的分析结论

[写作 Agent] → 负责将分析结论转化为可读的报告
              ↓ 最终输出结构清晰、表达流畅的研究报告
```

研究 agent 调用 web search 工具获取最新数据，分析 agent 调用 python 工具进行数据处理和统计，写作 agent 接收前两者的输出，专注于语言组织和叙事结构。每个 agent 都在自己最擅长的环节发力，最终的 Report 质量远高于单 Agent 全程包办。

更重要的是，这个流程是**可干预的**：如果你觉得分析深度不够，在分析 agent 输出后加一个"批判性审查 agent"；如果报告需要多语言版本，加一个"翻译 agent"。系统像乐高一样可拆卸、可扩展。

### 不是更复杂，而是更清晰

多 Agent 系统表面上看起来比单 Agent 复杂——有更多的组件、更多的交互、更多的状态需要管理。但这种复杂性是**良性的**。

它把混沌的"一个 agent 做所有事"拆解为清晰的"不同 agent 做不同事"，用结构化的方式处理复杂任务。每一个 agent 的职责边界明确，每一个 agent 的输入输出可观测、可验证、可干预。当系统出问题，你能快速定位到是哪个 agent 出了故障；当你需要优化，你能针对性地调整某个环节而不影响全局。

## 2. Hub and Spoke 架构

### 架构图示与核心原理

在构建多智能体（Multi-Agent）系统时，如何让一群 Agent 高效协作，是一个比实现单个 Agent 更有挑战的问题。Hub and Spoke 架构是其中一种被广泛采用的协作模式，它的核心思想很简单：**用中心节点统一协调，周围节点专注执行**。

想象一幅星形拓扑图：中间有一个 **Hub（轮毂）**，四周环绕着若干 **Spoke（辐条）**，每根辐条的末端连着一个专家节点。所有连线都从 Spoke 指向 Hub，Spoke 与 Spoke 之间**没有任何直接连线**。

```
          ┌─────────┐
          │ Spoke A │
          └────┬────┘
               │       ┌─────────┐
    ┌───────────┼──────▶│  Hub    │
    │           │       └───┬─────┘
    │           │           │
┌───┴───┐       │     ┌─────┴────┐
│Spoke B│       │     │  Spoke C │
└───────┘       │     └──────────┘
                
（Spoke 之间无直接连线，均通过 Hub 中转）
```

**核心原理**归纳为两点：

- **单向通信**：信息从 Spoke 发出后，必须经过 Hub 才能抵达其他 Spoke。
- **职责分离**：Hub 负责调度和整合，Spoke 负责执行特定任务，两者各司其职，互不越界。

### 协调者（Coordinator）的职责与核心能力

**Coordinator 是整个系统的中央枢纽**，它承担以下职责：

1. **接收任务**：从用户或其他系统入口接收原始需求。
2. **任务拆解**：将复杂任务拆分为多个可独立执行的子任务。
3. **分配任务**：判断每个子任务该交给哪个 Spoke，并下发执行指令。
4. **结果收集**：等待各个 Spoke 返回执行结果。
5. **结果验证**：检查返回内容是否符合预期格式和质量要求。
6. **结果组装**：将多个子结果整合为完整的最终答案或执行方案。

支撑这些职责，Coordinator 需要具备三项核心能力：

- **意图理解能力**：准确把握用户需求的真实意图，避免拆解方向跑偏。
- **调度决策能力**：知道哪个专家擅长处理哪类问题，合理分配任务。
- **上下文管理能力**：在收集和组装阶段，能正确拼接来自不同来源的信息片段，保持全局一致性。

### 专家（Specialist）的职责定义

Spoke 节点代表的是**领域专家**，每个 Spoke 只专注于做**一件事**，并且把这件事做到极致。

例如，在一个客服多智能体系统中，可能存在以下 Spoke：

- **商品查询专家**：只负责回答商品库存、规格、价格等问题。
- **物流追踪专家**：只负责查询订单物流状态。
- **退换货专家**：只处理退换货申请和进度查询。

这种设计带来了几个直接好处：

- **专业深度**：由于任务范围收窄，专家的回答质量更容易保持高水平。
- **可维护性**：修改一个专家的行为，不会意外波及另一个专家。
- **可扩展性**：增加新业务时，只需新增一个 Spoke，而无需改动现有专家。

### 为什么专家之间不能直接通信

这是 Hub and Spoke 与其他多智能体协作模式最核心的区别。**强制所有通信经过 Hub，是有意为之的设计约束**，而非技术限制。

**上下文隔离**是这个约束的核心目的。当 Spoke 之间不直接对话时，每个 Spoke 的上下文是**独立且可控**的。具体而言：

- **避免状态污染**：如果 Spoke A 可以直接给 Spoke B 发消息，B 的上下文会被 A 的内容污染。当系统规模扩大时，这种交叉影响会导致不可预测的行为。
- **防止循环依赖**：想象一个场景——A 需要 B 的结果，B 需要 C 的结果，C 又需要 A 的结果。如果允许直接通信，系统很容易陷入死锁或无限循环。强制经过 Hub，Coordinator 可以主动检测并打破这类循环。
- **简化调试和追踪**：所有信息流都经过同一个节点，日志记录和故障排查都变得清晰可控。你只需要看 Hub 的通信记录，就能还原完整的交互链路。
- **保持单一决策点**：没有 Hub 的协调，如果多个 Spoke 之间形成网状通信，系统的决策逻辑会分散到各处，难以保证全局一致的行为。

### 与"全能型 Agent"的对比

在讨论多智能体架构时，一个自然的疑问是：**为什么不直接训练一个超级 Agent，让它具备所有能力？**

这种"全能型 Agent"思路在工程实践中会遇到几个难以逾越的瓶颈：

| 维度 | 全能型 Agent | Hub and Spoke |
|------|-------------|--------------|
| **任务处理** | 一次处理所有子任务，上下文窗口压力巨大 | 每个 Spoke 只处理一个子任务，上下文相互隔离 |
| **输出稳定性** | 多个任务域混在一起，模型容易产生"角色混淆" | 每个 Spoke 有固定职责，行为边界清晰 |
| **系统扩展** | 增加新能力需要重新训练或大幅微调整个模型 | 增加新 Spoke 即可，不影响现有节点 |
| **故障隔离** | 一个模块出错可能影响全局输出 | 单个 Spoke 故障只会影响对应子任务 |
| **可解释性** | 决策过程黑盒，难以追踪哪个步骤出了问题 | 通信链路清晰，每个环节可独立审计 |

根本的矛盾在于：**模型的能力是有上限的，而业务需求是无限扩展的。** 把所有功能塞进一个 Agent，本质上是在和这个上限博弈。Hub and Spoke 架构通过**分布式任务分解**，把复杂度从"模型内部"转移到了"系统架构层面"，用工程化的协作方式弥补了单一模型的能力边界。

## 3. 专家角色拆分原则

在构建多 Agent 协作系统时，最常见也最致命的错误只有两个：要么把所有任务塞进一个 Agent 导致它不堪重负，要么把简单任务拆得过细导致协作开销远超收益。本节聚焦**如何拆**——拆分的判断维度是什么、典型团队配置有哪些、以及什么时候坚决不拆。

### 拆分的三个判断维度

并不是所有任务都需要拆分成多个 Agent。在动手之前，先用三个维度做一次冷静的判断。

**能力类型是否不同** — 这是最核心的判断标准。如果一个任务涉及**研究信息**、**分析数据**和**写作输出**三种截然不同的认知活动，强行塞给一个 Agent 的结果往往是每样都做到 60 分，而不是每样都做到 90 分。

必须拆分的典型场景：
- 需要深度搜索 + 需要逻辑分析 + 需要流畅写作 → 三个不同能力的 Agent
- 需要精确数学计算 + 需要自然语言解释 → 计算和解释是不同能力
- 需要实时工具调用 + 需要创意判断 → 工具型和创意型 Agent 天然不同

**输出格式是否不同** — 当一个 Agent 需要同时输出结构化数据（如 JSON）和自然语言（如报告）时，两种输出格式的生成逻辑完全不同。强行让单一 Agent 兼顾，往往在格式切换时出现质量滑坡。

**工具集是否不同** — 如果两个子任务需要调用的工具几乎没有重叠，拆分就是合理的选择。共享工具集反而会增加 Agent 的决策负担——它需要在每次执行时判断"这个任务该用哪个工具"。

> **经验法则**：三个维度中满足**任意一个**，就可以考虑拆分；满足**两个以上**，几乎一定要拆分。如果三个都不满足，那大概率是在过度设计。

### Content Team：内容生产流水线

Content Team 是最直观的团队配置，适合需要**深度研究 → 结构化输出 → 精细润色**的内容创作场景，比如技术博客，白皮书，营销文案。

| Agent | 职责 | Prompt 要点 |
|-------|------|-----------|
| **Researcher** | 从多来源收集信息，整理成结构化 notes | 明确可信度要求，指定整理格式，设定信息饱和度阈值 |
| **Outline Architect** | 基于研究素材设计文章结构 | 输入是 Researcher 的笔记，输出是分层大纲，标注核心论点 |
| **Draft Writer** | 依据大纲填充内容 | 严格遵循给定大纲，指定目标读者和语言风格 |
| **Editor** | 检查逻辑连贯性和信息准确性 | 给出具体修改指令，标记矛盾之处 |
| **Formatter** | 转化为发布标准格式 | 指定平台格式规范，输出可直接发布的终稿 |

### Customer Support Team：客服支持流水线

Customer Support Team 适合需要**快速分类 → 精准检索 → 专业回复**的场景，比如企业客服机器人和技术支持系统。

| Agent | 职责 | Prompt 要点 |
|-------|------|-----------|
| **Classifier** | 解读用户意图，分类并评估紧急程度 | 提供完整分类体系，输出结构化结果 |
| **Knowledge Base Searcher** | 在知识库中精准匹配相关文档 | 输入是分类标签 + 原始问题，输出最相关的 N 条知识条目 |
| **Response Drafter** | 综合问题和知识库内容撰写回复 | 指定回复风格和长度上限，知识不足时触发人工介入 |
| **Quality Checker** | 审核准确性、完整性和语气 | 检查关键信息与知识库一致，输出通过/需修改 |

### Business Analysis Team：商业分析流水线

Business Analysis Team 面向**数据驱动决策**场景，适合投资分析、市场研究、经营诊断等需要从原始数据提炼行动建议的工作。

| Agent | 职责 | Prompt 要点 |
|-------|------|-----------|
| **Data Collector** | 从多个数据源拉取并清洗原始数据 | 明确字段、时间范围、粒度，输出标准化中间格式 |
| **Trend Analyst** | 识别趋势、周期和拐点 | 标注统计显著性和置信区间，区分噪声和真实信号 |
| **Risk Assessor** | 识别潜在风险，评估影响和概率 | 提供风险分类框架，量化风险评分 |
| **Recommendation Writer** | 将洞察转化为可操作建议 | 每条建议必须有数据支撑，区分短期和长期战略 |

### 三个团队配置横向对比

| 维度 | Content Team | Customer Support Team | Business Analysis Team |
|------|-------------|----------------------|----------------------|
| **Agent 数量** | 5 | 4 | 4 |
| **核心能力** | 研究 / 写作 / 编辑 | 分类 / 检索 / 审核 | 数据 / 分析 / 建议 |
| **流水线类型** | 线性串行 | 线性串行 + 质量门禁 | 线性串行 |
| **典型工具集** | 搜索、文档编辑、格式转换 | 知识库 API、分类模型、消息推送 | 数据库、BI 工具、报表生成 |
| **实时性要求** | 低（批处理为主） | 高（毫秒级响应） | 中（分析任务可异步） |
| **适用场景** | 深度内容创作、白皮书 | 客服机器人、技术支持 | 投资分析、市场研究 |

### 什么时候不该拆

拆分是有成本的。以下情况，请坚决**不拆**：

- **简单查询类任务**：天气查询、单位换算、术语解释，一个 Agent 配合适的工具集，10 行 Prompt 就能搞定
- **输出是单一格式且流程线性**：如果任务是从 A 到 B 的线性流程，强行拆成三段只会增加协调成本
- **协作收益低于通信开销**：任务规模较小时，多 Agent 协作的 token 消耗和延迟会显著高于单 Agent 直接完成

**实战经验**：先跑通单 Agent 版本，确认任务确实需要多种能力后再拆。设计多 Agent 协作的正确姿势是**先假设不拆，只有当不拆明显有问题时再动手**。

## 4. 三个常见失败模式

即使 prompt 写得漂亮、角色定义清晰，系统仍然会在几个微妙的地方悄然崩溃。以下是三个最常见的失败模式。

### 1. Narrow Decomposition：拆解太窄，遗漏半壁江山

**问题**：协调者在拆解任务时，只覆盖了它认为重要的子主题，而遗漏了实际上不可或缺的其他领域。

**具体例子**：写一篇"2024年大语言模型技术进展"的技术博客，协调者拆解为：GPT-4o 进展 + Claude 3.5 进展 + 综合报告，但实际上遗漏了开源模型（Llama 3、Mixtral）以及多模态能力的实际落地情况。

**预防方法**：在拆解之前，**先枚举完整的问题空间，再拆解**。让协调者先生成一个完整的范围清单，确保没有任何重要领域被留空。

### 2. Lost Context：上下文丢失

**问题**：研究 Agent 挖掘出的重要发现，因为协调者没有显式地把这些发现写入写作 Agent 的 prompt，导致写作 Agent 对研究结果一无所知。

**具体例子**：研究 Agent 发现了三个关键数据点，但协调者传递给写作 Agent 的 prompt 只写了"请写一篇关于 LLM 技术进展的博客文章"，写作 Agent 完全不知道这三个发现，最终文章变成泛泛而谈。

**预防方法**：上下文必须**显式包含在每个 prompt 里**。每当你调用一个子 Agent，必须在它的 prompt 中完整写出它需要知道的所有信息，而不是依赖任何隐式的共享状态。

### 3. Telephone Effect：信息逐级失真

**问题**：当信息从协调者一路传递到最终 Agent 时，每一层都会对信息进行"优化"——删掉它认为不重要的细节、用自己的语言重新表述。到了最终 Agent 那里，原始信息已经面目全非。

**具体例子**：协调者原始指令包含精确的 benchmark 数据和成本影响，经过研究者 A → 研究者 B 两层传递，最终写作 Agent 完全不知道原始指令中的精确数据。

**预防方法**：在每个阶段都包含**原始数据**，而不仅仅是上一个 Agent 的输出。写作 Agent 的 prompt 里应该同时包含：协调者的原始指令 + 研究者的输出 + 原始参考资料链接。

### 关键规则：显式传递是唯一的信任来源

这三个失败模式有一个共同的根因：**多 Agent 系统中的信息不会自动流动**。协调者必须像对待一个分布式系统一样对待信息传递——每一个信息包都需要被显式序列化和反序列化。

**专家 Agent 不会自动继承协调者的上下文**。你必须把需要传递的信息完整地写入下一个 Agent 的 prompt 里。这不是最佳实践，这是必须遵守的工程纪律。

### 显式传递上下文的代码对比

```python
# ❌ 错误写法：不传递上下文
def write_report(report_type: str, style: str):
    """写作 Agent 的 prompt —— 没有包含研究结果"""
    return f"""
    You are a technical writer.
    Write a {report_type} report in {style} style.
    """


# ✅ 正确写法：显式传递上下文
def write_report(report_type: str, style: str, research_findings: dict, original_brief: str):
    """写作 Agent 的 prompt —— 包含完整上下文"""
    return f"""
    You are a technical writer.

    Original brief from the coordinator:
    {original_brief}

    Research findings to incorporate:
    {research_findings}

    Write a {report_type} report in {style} style.
    Make sure to include all specific data points from the research findings.
    """
```

第二个函数显式接收 `research_findings` 和 `original_brief` 两个参数，并将它们完整地嵌入 prompt。在多 Agent 系统中，**显式传递是信任的唯一来源**。

## 5. 上下文管理与失败处理

在多 Agent 架构中，随着任务规模扩大和专家数量增加，上下文无限增长、结果不可信、专家偶发失败等问题会迅速成为系统的致命瓶颈。

### 上下文窗口管理

#### 问题的本质

协调者调度专家时，每个专家完成任务后返回的内容会不断累积在协调者的上下文窗口中。一个典型的多阶段任务涉及十多个专家调用，如果每个专家返回几千 token 的自然语言描述，几轮迭代后上下文就会逼近模型上限。这种增长是**隐性的**——模型在上下文膨胀时不会报错，只会出现指令遵循质量下降这种难以定位的慢性故障。

#### 解法一：结构化输出

让专家返回结构化数据而非自然语言，是最直接的控制手段：

```python
from pydantic import BaseModel
from typing import List

class ExpertResponse(BaseModel):
    status: str  # 'success' | 'partial' | 'failed'
    summary: str  # 简短摘要，100字以内
    extracted_data: dict  # 结构化提取结果
    citations: List[str]  # 参考来源
```

结构化输出的核心收益是**信息密度提升**：原本一段 2000 字的 prose，压缩成几个字段后可能只剩 300 token，同时关键信息毫发无损。

#### 解法二：三层记忆架构

| 层级 | 名称 | 作用域 | 存储位置 |
|------|------|--------|----------|
| L1 | 工作记忆 | 当前任务当前阶段 | 内存 / 上下文 |
| L2 | 情景记忆 | 当前任务历史摘要 | Redis / 磁盘 |
| L3 | 长期记忆 | 跨任务事实与模式 | 向量数据库 / 文件 |

三层架构的设计哲学是：**上下文只保留决策所需的最小信息，详细历史交给外部存储**。协调者每完成一个阶段，就把"怎么做到的"写入 L2，上下文里只留"做到了什么"的概览。

#### 解法三：Checkpoint 落盘

对于超长流程任务，更保守的做法是**强制 Checkpoint 机制**——每个阶段结束时，协调者将当前状态序列化到磁盘。下游阶段如果发现上下文不足，直接从 Checkpoint 文件恢复，而不是把所有历史重新加载到内存。本质是**用存储换上下文**。

### 结果校验机制

协调者收到专家输出后，必须经过**三层校验**才可进入下一阶段：

1. **结构校验**：验证返回字段是否完整、类型是否匹配，通常由 Pydantic 或 JSON Schema 自动完成
2. **语义校验**：验证字段值是否满足业务约束，相关性评分是否合理
3. **质量门禁**：评估输出质量得分，低于阈值的输出直接打回重做

当校验失败时，系统会注入详细的错误反馈重新调用对应专家，并记录重试次数。设置最大重试上限（如 N=3），避免无限循环。

### 失败处理策略

**重试策略**采用指数退避（Exponential Backoff）。首次失败立即重试，第二次失败等 1 秒，第三次等 4 秒，以此类推。超过最大重试次数后，标记为永久失败。

**熔断机制**防止某个专家的持续故障拖垮整个系统。当一个专家在时间窗口 T 内失败次数超过阈值（如 5 次），触发熔断——后续请求在熔断期内直接返回降级结果。熔断期结束后放量一个请求试探，如果恢复则关闭熔断，否则继续熔断。

**三级降级**是更宏观的容错设计：

1. **专家 Agent 降级**：某个专家不可用时，使用更通用的大模型直接处理
2. **通用搜索降级**：LLM 幻觉风险高时，降级到规则引擎或搜索引擎获取确定性结果
3. **缓存数据降级**：连搜索都失败时，返回最近一次成功的缓存结果，并标注数据可能过期

**部分成功**策略用于并行执行场景。设定成功率阈值（如 80%），只要达标的专家数量足够，就继续推进流程，未成功的任务记录为异步后续处理。

## 总结

多 Agent 系统设计，本质上是在回答三个问题：

**1. 谁来协调？**
协调者是整个系统的核心，负责接收任务、拆解、分配、收集、验证和组装。协调者不是专家，专家之间也不直接通信——唯一的信息传递路径是协调者的上下文。

**2. 专家如何拆分？**
按"做什么事"拆，不按"怎么做"拆。能力类型不同、输出格式不同、工具集不重叠——满足任一条件就考虑拆分，满足两个以上几乎一定要拆。

**3. 工程上如何稳定运行？**
结构化输出控制信息密度，三层记忆架构管理上下文膨胀，三层校验守护质量，熔断降级策略提供韧性。

多 Agent 不是银弹，它的价值在于解决了一个单 Agent 无法绕过的根本问题：**专业化与多样性的矛盾**。用结构化的协作方式，让不同的 agent 做不同的事，最终产出超越任何一个 agent 单独能达到的质量。
